⏺ Here's my evaluation:

  fast-xml-parser vs Goessner xml2json — Honest Comparison

  fast-xml-parser

  Pros:
  - Actively maintained — v5.3.5 published 3 days ago (2026-02-08), consistent
  updates since 2017
  - 45M+ weekly downloads — massive ecosystem adoption (used by AWS SDK)
  - Zero DOM dependency — parses raw XML strings directly (works in Node,
  browser, Web Workers)
  - Configurable — attributeNamePrefix, ignoreAttributes, isArray callback (can
  solve the single-element array problem at parse time), preserveOrder, tag
  transforms
  - XMLBuilder included — bidirectional conversion in one package
  - TypeScript types built-in
  - XML validation built-in
  - 1 dependency (strnum)

  Cons:
  - https://advisories.gitlab.com/pkg/npm/fast-xml-parser/CVE-2026-25128/ — DoS
  via out-of-range numeric entities (fixed in v5.3.4)
  - https://github.com/RooCodeInc/Roo-Code/issues/7107 — unexpectedly decodes
  entities, causing mismatches
  - The single-element array problem is solvable via isArray config but requires
   you to enumerate which tags should always be arrays
  - Breaking changes between v4 → v5 (the project currently uses v4.3.3)

  Goessner xml2json/json2xml

  Pros:
  - Simple — ~200 lines of code, easy to understand and debug
  - Well-defined convention — the
  https://goessner.net/articles/xmljson/index.html is a widely-referenced
  standard for XML↔JSON mapping
  - No dependencies — zero external packages
  - Vendorable — can be dropped into @dde/util and fully owned/modified

  Cons:
  - Unmaintained — original is a static JS file from ~2006, no npm package
  - No TypeScript types — would need to write them yourself
  - DOM-dependent — requires DOMParser first (extra step, won't work in
  non-browser environments without polyfill like jsdom)
  - No isArray equivalent — same single-element array problem, no built-in
  solution
  - No XML builder validation — json2xml() doesn't validate output
  - No active community — no issue tracker, no bug fixes, no security patches
  - The npm forks (xml2json last published 6 years ago, p3x-xml2json is a C++
  node-gyp binding — not the same library)
  - @ prefix — would require changing every @_ reference across all existing DDI
   interfaces (100+ occurrences)

  Recommendation

  fast-xml-parser is the stronger choice for this project, for these reasons:

  1. The isArray option solves the biggest pain point — you can configure it
  once to always return arrays for var, varGrp, catgry, sumStat, eliminating the
   normalization guards entirely
  2. No DOM dependency — cleaner for testing (Vitest/Node), Web Workers, SSR
  3. Active maintenance — security patches within days (CVE fixed in v5.3.4 on
  Jan 30)
  4. v5 upgrade path — moving from v4.3.3 to v5.3.5 is a natural part of the
  rewrite, and v5 has better APIs
  5. Vendoring Goessner means owning all bugs forever — with no community to
  report/fix them

  The testing difficulties you mentioned — could those be the single-element
  array issue? If so, fast-xml-parser v5's isArray callback eliminates that
  entirely:

  const parser = new XMLParser({
    ignoreAttributes: false,
    attributeNamePrefix: '@_',
    isArray: (name) => ['var', 'varGrp', 'catgry', 'sumStat',
  'catValu'].includes(name),
  });

     │ State Management                                                        │
     │                                                                         │
     │ - No global NgRx Store. Each package has its own signalStore() provided │
     │  at component level.                                                    │
     │ - Host app has a DataverseStore (SignalStore) that holds canonical      │
     │ truth.                                                                  │
     │ - Data normalization happens in the host app (DdiNormalizer utility     │
     │ converts ddiJSONStructure → DdeVariable[] for reads).                   │
     │ - When package outputs fire, host app updates DataverseStore,           │
     │ re-normalizes, and updated inputs flow back to packages.                │
     │                                                                         │
     │ DDI XML Fidelity                                                        │
     │                                                                         │
     │ The raw ddiJSONStructure (as parsed by fast-xml-parser v5 with @_       │
     │ attribute prefixes) is always retained in the host app's                │
     │ DataverseStore. This is the single source of truth for uploads.         │
     │                                                                         │
     │ Read path (DDI → packages):                                             │
     │ ddiJSONStructure → DdiNormalizer.toVariables() → DdeVariable[] →        │
     │ package input                                                           │
     │ This is a lossy projection — packages only see the clean normalized     │
     │ view.                                                                   │
     │                                                                         │
     │ Write path (packages → DDI):                                            │
     │ package output { variableId, changes } →                                │
     │ DdiNormalizer.applyChanges(rawDdi, changes) → updated ddiJSONStructure  │
     │ applyChanges() mutates the raw DDI structure directly, preserving all   │
     │ @_ prefixed attributes, XML element ordering, and any fields the        │
     │ packages don't know about. The raw structure is what gets serialized    │
     │ back to XML via fast-xml-parser's XMLBuilder and uploaded to Dataverse. │
     │                                                                         │
     │ This ensures the uploaded XML is always DDI-spec compliant — the        │
     │ normalized models are never serialized to XML.                          │
     │                                                                         │
     │ XML Parser: fast-xml-parser v5                                          │
     │                                                                         │
     │ Upgrade from v4.3.3 to v5.x. Key configuration:                         │
     │                                                                         │
     │ import { XMLParser, XMLBuilder } from 'fast-xml-parser';                │
     │                                                                         │
     │ const parserOptions = {                                                 │
     │   ignoreAttributes: false,                                              │
     │   attributeNamePrefix: '@_',                                            │
     │   // Eliminate the single-element array problem at parse time           │
     │   isArray: (name: string) => [                                          │
     │     'var', 'varGrp', 'catgry', 'sumStat', 'catValu', 'notes'            │
     │   ].includes(name),                                                     │
     │ };                                                                      │
     │                                                                         │
     │ const parser = new XMLParser(parserOptions);                            │
     │ const builder = new XMLBuilder(parserOptions);                          │
     │                                                                         │
     │ The isArray callback ensures DDI elements that can appear multiple      │
     │ times are always parsed as arrays, even when only one element exists.   │
     │ This eliminates the class of bugs where Array.isArray() guards were     │
     │ needed throughout the codebase.                                         │

     Data Explorer Nx Monorepo Rewrite Plan

     Context

     The Dataverse Data Explorer is an Angular 17 single-page app for
     browsing, editing, and cross-tabulating research dataset metadata from
      Dataverse repositories. It currently uses jQuery PivotTable, global
     NgRx Store, PrimeNG, and is a monolithic app that can't be reused.

     The project needs a complete rewrite because:
     - The cross-tabulation calculator and variable editor must be
     embeddable as Web Components in other apps (odesi-rest Angular 14,
     plain HTML, iframes)
     - jQuery/PrimeNG must be removed
     - State management has critical bugs (identified in prior audit)
     stemming from NgRx boilerplate complexity
     - The app needs to move to Angular 21 + Nx 22.3 (zoneless, Vitest,
     signals-first)

     Target Architecture

     Nx 22.3 monorepo with Angular 21.1 (zoneless by default), containing:

     data-explorer-nx/
       apps/
         data-explorer/              # Main app — fetches from Dataverse,
     composes packages

       libs/
         shared/
           models/                   # @dde/models — pure TS interfaces (no
      Angular)
           util/                     # @dde/util — pure helper functions
           ui/                       # @dde/ui — shared Angular primitives
     (chart, i18n, theme, a11y)

         pivot-table/                # @dde/pivot-table — pure TS engine +
     Angular table renderer
         cross-tab-calculator/       # @dde/cross-tab-calculator — full
     cross-tab feature (Web Component)
         variable-editor/            # @dde/variable-editor — full variable
      editing feature (Web Component)

     Key rules:
     - Packages are data-agnostic — no HTTP calls, no Dataverse knowledge.
     Data flows in via input(), events flow out via output().
     - Packages use component-scoped SignalStore (not providedIn: 'root').
     - Packages accept translations as a flat object (no ngx-translate
     dependency).
     - Packages expose Web Component registration via @angular/elements +
     createCustomElement.
     - No PrimeNG — native HTML + pure CSS for all UI controls.
     - No jQuery — custom PivotEngine in @dde/pivot-table replaces jQuery
     PivotTable.
     - No Tailwind/DaisyUI — pure CSS with CSS custom properties for
     theming.
     - Zoneless by default (Angular 21). No Zone.js.
     - DDI XML fidelity — the raw ddiJSONStructure (with @_ prefixed
     attributes) is the source of truth for uploads. Normalized DdeVariable
      models are a read-only view layer. All mutations flow back through
     the raw DDI structure to ensure Dataverse receives spec-compliant XML.
     - fast-xml-parser v5 — upgrade from v4.3.3 to v5.x. Use the isArray
     callback to eliminate the single-element array normalization problem
     at parse time. Retain @_ attribute prefix and #text conventions.

     Dependency Graph

     @dde/models          → (none)
     @dde/util            → @dde/models
     @dde/ui              → @dde/models
     @dde/pivot-table     → @dde/models
     @dde/cross-tab-calculator → @dde/models, @dde/util, @dde/ui,
     @dde/pivot-table
     @dde/variable-editor      → @dde/models, @dde/util, @dde/ui
     data-explorer (app)       → all libs

     Clean interfaces that strip the DDI XML @_ prefix convention:

     interface DdeVariable {
       id: string;  name: string;  label: string;  interval: string;
       isWeight: boolean;  assignedWeightId: string;
       question: { literal: string; interviewer: string; post: string };
       universe: string;  notes: string;
       categories: DdeCategory[];  summaryStatistics: DdeSummaryStatistics;
       format: string;
     }

     interface DdeVariableGroup {
       id: string;  label: string;  variableIds: string[];
     }

     interface DdeCategory {
       value: number;  label: string;  frequency: number;
     weightedFrequency: number;
     }

     type CrossTabObservationData = { [variableId: string]: string[] };
     type DdeTranslations = { [key: string]: string };
     type DdeLocale = 'en-CA' | 'fr-CA';
     type DdeTheme = 'light' | 'dark';

     State Management

     - No global NgRx Store. Each package has its own signalStore()
     provided at component level.
     - Host app has a DataverseStore (SignalStore) that holds canonical
     truth.
     - Data normalization happens in the host app (DdiNormalizer utility
     converts ddiJSONStructure → DdeVariable[] for reads).
     - When package outputs fire, host app updates DataverseStore,
     re-normalizes, and updated inputs flow back to packages.

     DDI XML Fidelity

     The raw ddiJSONStructure (as parsed by fast-xml-parser v5 with @_
     attribute prefixes) is always retained in the host app's
     DataverseStore. This is the single source of truth for uploads.

     Read path (DDI → packages):
     ddiJSONStructure → DdiNormalizer.toVariables() → DdeVariable[] →
     package input
     This is a lossy projection — packages only see the clean normalized
     view.

     Write path (packages → DDI):
     package output { variableId, changes } →
     DdiNormalizer.applyChanges(rawDdi, changes) → updated ddiJSONStructure
     applyChanges() mutates the raw DDI structure directly, preserving all
     @_ prefixed attributes, XML element ordering, and any fields the
     packages don't know about. The raw structure is what gets serialized
     back to XML via fast-xml-parser's XMLBuilder and uploaded to
     Dataverse.

     This ensures the uploaded XML is always DDI-spec compliant — the
     normalized models are never serialized to XML.

     XML Parser: fast-xml-parser v5

     Upgrade from v4.3.3 to v5.x. Key configuration:

     import { XMLParser, XMLBuilder } from 'fast-xml-parser';

     const parserOptions = {
       ignoreAttributes: false,
       attributeNamePrefix: '@_',
       // Eliminate the single-element array problem at parse time
       isArray: (name: string) => [
         'var', 'varGrp', 'catgry', 'sumStat', 'catValu', 'notes'
       ].includes(name),
     };

     const parser = new XMLParser(parserOptions);
     const builder = new XMLBuilder(parserOptions);

     The isArray callback ensures DDI elements that can appear multiple
     times are always parsed as arrays, even when only one element exists.
     This eliminates the class of bugs where Array.isArray() guards were
     needed throughout the codebase.

     i18n Strategy

     Two-layer approach: build-time for the host app, runtime for packages.

     Host App: @angular/localize (build-time)

     - Uses $localize tagged templates for its own strings (header, loading
      states, error messages, tab labels)
     - One build per locale, deployed at /en-CA/ and /fr-CA/ paths
     - Language switch triggers a page redirect to the other locale's build
      (no runtime switching needed)
     - Translation files: messages.en-CA.xlf, messages.fr-CA.xlf (standard
     Angular i18n extract/merge workflow)
     - ng extract-i18n extracts host app strings; translators maintain XLF
     files

     Packages: Signal-based translate pipe (runtime)

     - Packages do NOT use @angular/localize or ngx-translate internally
     - Packages accept a translations: DdeTranslations input — a flat { 
     [key: string]: string } map
     - @dde/ui provides:
       - DDE_TRANSLATIONS injection token (writable signal)
       - DdeTranslatePipe — standalone pipe that reads from the signal and
     returns translated strings
       - DdeTranslateDirective — for attribute-based translation (e.g.,
     [ddeTranslate]="'KEY'")
     - Each package's root component syncs its translations input into the
     signal via effect()

     How the host app bridges both layers

     - The host app maintains package translation JSON files separately:
     package-translations.en-CA.json, package-translations.fr-CA.json
     - These contain all keys used by @dde/cross-tab-calculator and
     @dde/variable-editor
     - At build time, the correct locale's package JSON is included in the
     build assets
     - The host app loads the JSON at startup and passes it to packages via
      translations input
     - Since the host app is already locale-specific (one build per
     locale), it knows which JSON to load

     How external consumers (odesi-rest, plain HTML) provide translations

     - The consuming app provides its own translation map directly:
     <dde-cross-tab-calculator translations='{"ADD_VARIABLE": "Add 
     Variable", ...}'>
     - Or loads a JSON file and sets the property:
     fetch('/i18n/cross-tab.en.json').then(r => r.json()).then(t => {
       document.querySelector('dde-cross-tab-calculator').translations = t;
     });

     Translation key ownership

     - Host app keys: prefixed with nothing (e.g., HEADER.TITLE,
     ERROR.FETCH_FAILED) — managed via $localize + XLF
     - Cross-tab calculator keys: prefixed with CROSS_TAB. (e.g.,
     CROSS_TAB.ADD_VARIABLE) — managed via JSON
     - Variable editor keys: prefixed with VAR_EDITOR. (e.g.,
     VAR_EDITOR.SAVE) — managed via JSON
     - Shared UI keys: prefixed with SHARED. (e.g., SHARED.CANCEL,
     SHARED.CLOSE) — managed via JSON, included in both package JSONs

     DdeTranslatePipe implementation (in @dde/ui)

     // Injection token — writable signal so packages can update it from 
     their input
     export const DDE_TRANSLATIONS = new
     InjectionToken<WritableSignal<DdeTranslations>>('DDE_TRANSLATIONS');

     @Pipe({ name: 'ddeTranslate', standalone: true })
     export class DdeTranslatePipe implements PipeTransform {
       private translations = inject(DDE_TRANSLATIONS);

       transform(key: string): string {
         return this.translations()[key] ?? key;
       }
     }

     Package root component wiring

     @Component({
       providers: [{ provide: DDE_TRANSLATIONS, useFactory: () =>
     signal({}) }],

     export class CrossTabCalculatorComponent {
